# Security design

The public reporting policy is in [SECURITY.md](../SECURITY.md).

OpenClaw Quick Replies has three trust boundaries. First, evaluator output is untrusted: it is accepted only for explicit asks and must pass confidence, shape, complete-option-set, label-length, UTF-8 byte-length, and callback encoding checks. Second, Telegram callbacks are untrusted: only canonical `oqr:v1:<base64url>` payloads from authorized senders with source message identity and text can submit an inbound turn. Third, registry metadata and update callbacks are untrusted: only stable versions newer than the running plugin and matching a recent plugin-issued prompt can reach the native updater.

The managed evaluation run disables tools and message delivery, uses an isolated temporary session, and inherits OpenClaw's model and credential policy. The optional update checker makes a daily public metadata request to npm and stores bounded version/timestamp state under the OpenClaw state directory. It invokes OpenClaw with `execFile`, never a shell, force-reinstalls npm and ClawHub packages at the exact approved version, preserves and re-verifies the managed source and installed version, and requires separate authorized install and restart callbacks. Callback diagnostics contain actions, versions, and rejection or failure reasons, not callback payloads or conversation content.

A five-minute in-memory cache reduces duplicate model calls. Cache keys include message text, channel, the explicit evaluator model or current resolved host default model, and all behaviorally relevant configuration. A separate bounded five-minute map prevents repeated callbacks for the same Telegram source message. Neither is persisted or sent to a plugin-owned service; update state contains no conversation content or user identifiers.
